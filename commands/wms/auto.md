# /wms:auto — 智能超级命令

> 一个入口统筹任务检索、路线设计、执行验收与知识沉淀。

## 核心原则

1. **一个入口** -- 所有复杂任务先进入 `/wms:auto`，避免入口分散导致的上下文割裂。
2. **索引模式** -- Skill 优先走 `SkillIndexer` 索引摘要，而不是直接全量读取技能正文。
3. **长对话自动压缩** -- 长链路会话先检查 `compressContext`，必要时执行上下文压缩再继续。
4. **智能缓存** -- 已确认的能力索引、检索结果和阶段结论优先复用，减少重复扫描。
5. **统筹设计** -- 先形成任务地图、依赖关系和边界，再进入编码或排障。
6. **按规模执行** -- 小任务走最短闭环，大任务必须按阶段推进并保留检查点。
7. **原子化验收** -- 每次只推进一个明确目标，完成后立即验证，不把风险拖到最后。
8. **风格继承** -- 输出遵循项目现有命令体系、目录习惯和命名风格，不引入额外语义分叉。
9. **动态追加** -- 中途发现新的约束、依赖或副任务时，可在当前阶段补充而不是重开流程。
10. **可回溯** -- 每个关键决策都要能回溯到输入、代码位置和验证结果。
11. **知识沉淀** -- 任务完成后提炼为规则、经验或索引更新，避免重复踩坑。

## HARD CONSTRAINTS

- 约束 1：顺序锁定。必须按 Phase 顺序推进，当前阶段未通过验证前不得跳到下一阶段。
- 约束 2：quest-designer 必须被调用。中高复杂度任务需要先由 `quest-designer` 生成任务地图和执行路线。
- 约束 3：代码修改前置条件。未完成上下文检索、边界确认和 TodoWrite 拆解前，不进入文件修改。
- 约束 4：无简单任务豁免。即使任务很小，也至少要完成规模判断、边界确认和最小验证。

## PHASE 1: 上下文检索

### 1.0a 长对话自动压缩

先检查会话长度和决策密度，避免把历史噪音带入当前任务。

```
Read("src/utils.js")
  -> 定位 compressContext 与 CONTEXT_COMPRESSION
  -> 检查 CONTEXT_COMPRESSION.MESSAGE_THRESHOLD
  -> 若消息量超过 MESSAGE_THRESHOLD，则执行上下文压缩

TodoWrite("上下文压缩: pending|completed")
```

### 1.1 命令、Agent、插件检索

优先获取可用入口和自动化能力的轮廓，不做无差别全文加载。

```
Glob("$HOME/.claude/commands/wms/*.md")
Glob("$HOME/.claude/agents/*.md")
Glob("$HOME/.claude/plugins/**/*.md")
```

### 1.2 Skill 索引摘要

Skill 不再直接全量扫描 `skills` 目录。改为通过 `src/skills/skill-indexer.js` 中的 `SkillIndexer` 获取摘要，先看索引，再按需加载。

```
Read("src/skills/skill-indexer.js")
  -> 初始化 SkillIndexer
  -> 调用 getIndexSummary()
  -> 仅在摘要命中后调用 loadContent(relativePath)
```

### 1.3 代码与约束补全

围绕目标读取最少但完整的定义、测试和相邻实现，补齐任务边界。

```
Grep("关键词", "src/**/*.js")
Read("相关测试")
Read("相邻命令或能力定义")
```

### 1.4 输出检索结论

形成一份最小可执行结论，至少包含：目标文件、现有依赖、缺失项、风险点、是否需要跨服务或跨模块协作。

## PHASE 2: 任务设计

### 2.1 规模判断

- 小任务：单点修改或单文件修复，直接生成最短执行路线。
- 中任务：多文件协作、需兼容现有契约，先拆任务再执行。
- 大任务：需要架构决策、跨域联动或多轮验证，必须升级为完整 quest map。

### 2.2 生成任务地图

调用 `quest-designer` 输出 step-by-step 路线，至少包含：

- 目标与非目标
- 涉及文件与职责
- 风险与验证方式
- 先做什么、后做什么、何时停下来汇报

### 2.3 锁定执行边界

把“不该改什么”写清楚。若任务只是恢复命令定义，就不改 CLI、路由器或安装器；若只是修测试契约，就不顺手重构无关模块。

## PHASE 3: 原型与方案收敛

### 3.1 先拿脏原型

对文档型、命令型或逻辑型任务，都可以先形成一个最小原型，再对照项目风格压缩为最终版本。

### 3.2 交叉检查

原型至少要对齐以下来源：

- 直接测试契约
- 现有命令文档风格
- 相关源码导出名和实际文件路径

### 3.3 形成最终实施稿

将原型中的冗余描述、重复步骤和不必要的兜底逻辑去掉，只保留项目当前真正需要的协议。

## PHASE 4: 编码实施

### 4.1 最小作用域改动

只修改当前需求要求的文件或实现，避免把文案整理、架构重命名、历史 TODO 清理混进同一次交付。

### 4.2 与现有风格对齐

- 沿用当前命令目录的 frontmatter 结构
- 保持中文协议化表达
- 用已有文件路径和导出名，不发明别名

### 4.3 即改即校验

每次实质性修改后，立刻跑最相关测试或检查命令；若失败，优先修本次引入的问题，而不是扩大扫描范围。

## PHASE 5: 原子化验收

### 5.1 直接验证

优先执行与变更文件直接相关的测试、快照或解析校验。

### 5.2 回归边界检查

确认没有破坏以下链路：

- 命令元数据可被 catalog 正常解析
- 被引用的路径和导出符号真实存在
- 关键约束文字仍能被测试命中

### 5.3 未通过时止损

如果当前阶段验证失败，留在本阶段修复；不要带着已知红灯进入交付总结。

## PHASE 6: 交付与沉淀

### 6.1 交付摘要

向用户报告：改了什么、为什么这么改、验证了什么、还有什么未做。

### 6.2 沉淀知识

若这次任务暴露出新的通用模式，应补充到技能索引、工作流规则或项目记忆中。

### 6.3 输出后续动作

仅在确有必要时给出下一步，例如补跑更大范围测试、提交变更或扩展命令能力。
